## 4. Beyond the suite: render-level verification of DASB bank delivery

The suite pins the stamped banks via the $7E:E040 debug mirrors (with the rotate64 zero-mirror control). The mirrors are written one instruction after the hardware `sta $4307,x` from the same A value — so a wrong **value** cannot hide, but a wrong **channel offset** (CHX) could keep mirrors green while the hardware fetches a stale slice. I closed that gap on the framebuffer:

- On the 256 `_rotfreeze` build (positions frozen, headings advance), band-1's rendered content must be a function of h1 with **period 256, not 64** (only h&63 feeds the pointer; the slice index lives in the bank byte).
- Measured (within-run temporal hops, phase-safe): band-1 pixels at same-(h&63) hops **+64: 17,058 differ; +192: 17,063 differ** (slice-scale change — the bank byte demonstrably reaches the fetching channel); **full +256 wrap: 0 differ** (byte-identical, period-256 confirmed; +128 shows only ~1.8k px — the 180° view of the checker world is near-symmetric, so that hop is non-discriminating).
- Calibration: adjacent captured frames differ by ~1,881 band-1 pixels = the one-pose visual step — direct rendered evidence that band content changes at the pose cadence during a sustained turn (the DoD's rendered half at the finest granularity the capture path resolves).
- Harness note (confirms the trial's rig lesson rather than contradicting it): screenshot capture under frame-stepping carries a nondeterministic 0-or-1-frame lag (a first-pass +256 wrap showed a 1,861-px "mismatch" — exactly single-pose scale; re-measured with neighbours, wrap = 0). WRAM traces are immune; the suite's WRAM-for-cadence / hops-for-render split is the correct instrument choice.

## 5. VBlank-timing hazard review (the one NEW hazard class this sprint introduces)

HDMA reads the indirect data bank ($43x7) per line fetch mid-frame, so a DASB write during active display tears. Source review of **every** $4307 write site in the tree (grep over template + engine + macros — exactly six sites, all in this template):

- `set_indirect_banks` (main.asm:905, 915) — called only from `bind_matrix_pair`, boot path, **forced blank**.
- `stamp_pose_banks` (main.asm:1139, 1150, 1169, 1180) — called at main.asm:524 (boot, **forced blank**, before screen-on at :571-576) and main.asm:785 (per frame, **post-`wai`**, on the same code path and in the same VBlank window as the proven `stamp_origins` re-stamp at :788; the pointer rewrites at :736-783 precede it in the same window).

Envelope: the whole post-`wai` block for the 256 build (counter + headings + 4-axis 8.8 movement + 2×pointer math + 4×DASB stamp + origin re-stamp) statically counts to roughly 750 CPU cycles ≈ 4-5 scanlines against a ~38-line VBlank — ~8× margin. Empirical corroboration: 80-frame +1/+1 lockstep, 0-differing-row static equality, and no row-level artifacts in any full-frame diff. **No DASB write can land during active display on any shipped path.** (LATCH_VIOLATION's deliberate mid-display spin targets $210D, not DASB, and is the negative-control build.)
