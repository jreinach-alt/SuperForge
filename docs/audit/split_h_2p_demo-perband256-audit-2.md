### D2 (LOW — DASB bank binding tested via $E040 mirrors, a proxy): REMEDIATION VERIFIED

Audit-1's recommended hardening: "port the period-256/period-64 render discriminator into the suite (rotfreeze, same-(h&63) hop pair + wrap pair)." Shipped as `test_rot256_bank_delivery_render_period` (`tests/test_split_h_2p_demo.py:590-660`), and it implements exactly that shape. I verified it three ways:

1. **Ran it** — passes in-suite and 2× isolated (§1).
2. **Adversarial logic read** — the failure-mode claim holds. On the 256 rotfreeze build only `(h & 63)` feeds the pose pointer; the slice index travels ONLY in the stamped bank byte. If a wrong CHX offset kept the debug mirrors green while the hardware channel held a stale bank, rendered content would be a function of `h` with period 64 — so the +64-frame and +192-frame same-(h&63) hops would read only capture-lag noise. The assertions `d64 > 8000` / `d192 > 8000` (test lines 640-646) would then FAIL: the noise ceiling is ~4.7k px (measured, below) and a zero-lag broken path reads ~0. Both bands are asserted (the `for name, (y0, y1) in (("band1", …), ("band2", …))` loop covers PPU lines 0..111 and 112..223 with the +7 screenshot offset modeled) — necessary, since each pair has its own CHX homes. WRAM heading reads pin every hop's arithmetic (`(h - h_ref) & 0xFF == 64/192/0/0`), lag-free.
3. **Instrumented re-measurement** (my own probe mirroring the test, magnitudes printed):

```
headings: ref=56 +64->64 +192->192 wrap256->0 wrap512->0
band1: d64=25242 d192=28364 | wrap pairwise: ref-w256=4706 ref-w512=4706 w256-w512=0 min=0
band2: d64=25157 d192=28411 | wrap pairwise: ref-w256=4674 ref-w512=4674 w256-w512=0 min=0
```

   - Slice-scale hops measure ~25.2k/28.4k px per band — the remediation agent's "~25k" claim re-measured true, with >3× margin above the 8,000 bar (audit-1's own §4 measurement was ~17k on different hop framing; both are unambiguously slice-scale).
   - The threshold is well-placed: one-pose capture-lag noise ≈ 4.7k px << 8,000 << 25k. A period-64 render cannot clear the bar; a real slice change cannot miss it.
   - **The wrap-identity min-over-3-pairwise design was demonstrated live in my probe**: the `ref` capture happened to carry a different 0/1-frame lag than `w256`/`w512` — both ref-pairs read ~4.7k px (exactly the one-pose scale the remediation commit message reports from its own probe), while the same-lag pair `w256-w512` diffed **exactly 0**. A naive single-capture `wrap == 0` assertion would have flaked right here; the pigeonhole over three wrap-aligned captures (lag ∈ {0,1} → two captures must share a lag → an exact-0 pair exists) converts that hazard into a deterministic identity check, exactly as described. The wrap-identity also serves as the non-vacuity pairing (the metric provably reads zero through the same capture path).

   Residual note (out of the finding's scope, recorded for completeness, no action requested): the discriminator proves the stamped banks reach *fetching* channels with the correct period; a hypothetical swap of two *correct* per-band values across the bands' channel pairs would still pass this test (both bands would re-render at slice scale). That failure mode was not part of audit-1's D2 (which was about a stale slice behind green mirrors) and is partially constrained elsewhere (the static perband builds' byte-identity gate pins per-band content at boot bindings). The shipped test matches the recommendation as written.
